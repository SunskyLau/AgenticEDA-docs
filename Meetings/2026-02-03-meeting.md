## 2) 需要改进/补齐的能力（按优先级排序）

## P0（必须做，直接决定下周能不能交付）
### P0-A. Insight Map：从“语义散点”转向“可解释坐标系/格化表达”
**问题诊断**
- 仅靠语义 embedding 做 2D 聚类很难形成稳定簇；insight 往往“语义不相似但逻辑上串联”。

**会议讨论过的可行方案（都值得保留，先做最可落地的）**
1) **基于“故事线/链路（branch / chain）”的聚合**
   - 同一条 exploration chain 的节点空间上更靠近 → 自然形成 story cluster
   - 适合表达“这是一条推理线”，而不是“这是一堆语义近邻”
2) **显式 X/Y 轴编码（推荐优先实现，利于表达空白空间）**
   - 目标：让“未探索空间”可见（空格=没覆盖到的组合/类型）
   - 示例轴（会议提过的方向）：
     - X：数据列/属性（或属性组合；一个 insight 落在哪些列上）
     - Y：insight taxonomy（2 大类 + 9 小类）
   - 先用简单规则落格：insight → (列集合, 类型) → 位置；后续再优化聚合/密度编码
3) **混合布局（备选）**
   - 大框架格化（显示覆盖与空白），格子内部再用小范围力导向/堆叠表示 story branch 或时间顺序

**本周要达成的“最小可用版本（MVP）”**
- 先实现 **方案 2（轴编码格化）** 的 MVP：
  - 能把 insight 放到一个稳定位置
  - 能明显看出哪些区域“还没被探索”
  - 支持点击格子/点 → 打开 inspector → 看证据与来源任务

### P0-B. Scenario 驱动设计：先定 1–2 个“用户故事”，再反推界面与交互
**会议结论**
- 不先做 scenario，UI 和 map 的合理性无法验证，最终会和真实使用偏离。

**本周要产出的 scenario 结构（写成可复用模板）**
- 数据集背景（用户是谁/任务是什么）
- 初始进入系统 → 看见什么（data profile + 初始地图）
- 第一次产生兴趣点：用户为什么点它（motivation）
- 下钻路径：用户如何沿 chain 继续（drill-down vs 跳回根部换方向）
- 关键交互点：如何筛选/对比/回溯证据
- 输出：最终用户拿到什么（report / insight 集合 / 可分享结果）

---

### P0-C. UI Layout 收敛：VS Code 式“中间大图 + 两侧轻面板 + 底部过程”
**会议共识（必须落实到实现/原型）**
- **中间：核心可视化（insight map）**是主舞台，尽可能大
- **左侧：数据入口与数据概览**（upload + 简要 profile），避免堆太多统计图
- **右侧：inspector（按需出现/可折叠）**
- **底部：过程与状态**（history tree + agent/task 当前在做什么）

**具体交互/布局改动任务**
- Data profile：建议固定在左上（符合用户视线与流程）
- Data table：做成可展开/按需查看，默认不占核心区域
- Inspector：默认隐藏或抽屉式出现（点击点/格子/节点时弹出）
- Agent 状态：
  - 不一定需要独立大面板；可与 history 融合（例如高亮“正在探索”的节点/链路）
  - 保留轻量列表：queued / running / completed（用于 debug 与可解释）

## P1（重要但可延后；P0 跑通后再强化）
### P1-A. Insight 结构化与 taxonomy 标注：让可视化“直接消费”模型输出
**方向**
- 引入“insight taxonomy（2 大类 + 9 小类）”作为统一坐标/筛选维度
- 结构化粒度要更细：把“大 insight”拆碎，才能稳定落格/聚合

**两种落地策略（会议讨论过）**
1) 运行期结构化：analysis agent 输出时就带 label/结构（更一致，但可能束缚自由探索）
2) 总结期结构化（当前倾向）：insight summarization agent 负责拆分 + 分类 + 结构化（更灵活）

**建议的折中实现**
- 先用总结期结构化跑通（成本低）
- 后续对“高价值任务类型”再逐步加运行期约束

---

### P1-B. 探索调度策略：DFS/BFS/混合，以及用户控制的可用性
**问题**
- 纯 DFS/BFS 不符合真实分析（人会混合：下钻 + 回到根部换方向）
- 节点增长太快，用户不可能逐个指定“下一步做哪个”

**当前临时 heuristic（可保留作为 baseline）**
- 按“分析覆盖的数据范围/影响力”排序优先级：覆盖越大越先跑（保证系统持续推进）

**后续可实验的改进点**
- 加入 novelty / diversity：避免一直在相似区域重复探索
- 人在环控制：允许用户对某条 chain “加速/暂停/终止/置顶”
- 资源预算：并行 agent 数与队列长度的动态调度（但别把控制做得太复杂）

---

### P1-C. Report 生成：把 chain 变成可分享的 story
- 在任一节点/chain 上一键生成 report
- report 要能引用证据、说明动机、串联推理步骤（服务于对外沟通与论文叙事）

## 4) 我们需要共同对齐/拍板的关键决策（避免反复返工）

## 5) 下一次同步前的验收标准（我们拿什么来证明“做对了”）
- 能用一个 scenario 顺畅讲完：
  - 用户看到地图 → 发现空白/覆盖 → 点选某个区域 → inspector 显示证据与来源 → 下钻/回溯 → 生成简版 report
- 外部合作者能回答三个问题：
  1) 这个系统在分析什么？每个节点做了什么？
  2) insight 是什么样的产物？证据在哪里？
  3) 我能怎么用它继续探索或总结结论？
