# 核心目标与当前里程碑 (Milestone)

**项目核心定位**：提供一种交互与可视化空间（Interaction & Visualization Space），让用户能够监控（Monitor）并介入控制（Steer）一个长时运行、具有高度自主性的数据分析大模型智能体（Long-running Autonomous Data Analysis Agent）。 **当前最高优先级**：**收敛可视化设计**。Prof. Weng 明确指出，本轮迭代后应停止引入新的可视化形式，专注解决可扩展性（Scalability）问题，并系统化交互设计（Interaction Design），随后全力推进 Scenario 的构建和论文撰写。

## 任务分解一：前端可视化与布局优化 (Visualization & Layout)

**目标**：彻底解决当前 Storyline 过于稀疏、信息过载、画布空间爆炸以及缩放交互繁琐（Scalability）的问题，将视觉重心回归到“呈现 Agent 分析模式的全局演化态势”。

- $$$$
    
    **1. 流式布局与单向视点控制 (Streaming Layout & 1D Viewport Control)**：
    
    - **痛点**：当前 X 轴和 Y 轴双向滚动导致界面沦为“无边界画布”，用户极易迷失（Lost in space），且频繁缩放导致交互成本过高。
        
    - **动作**：强制将 Y 轴固定在“一屏（One Screen）”以内，仅保留 X 轴（时间线）的水平滚动能力。
        
    - **逻辑（学术表述）**：由于 Agent 的分析是一个基于时间序列的流式生成过程（Streaming process），系统的视点（Viewport）默认应自动向右滚动，**永远让用户关注最新生成的区域**。
        
    - **机制**：提供一个“暂停（Pause）”机制。当用户想要回溯之前的分析时，可以暂停自动滚动，手动向左滑动查看，完成检视或干预后再恢复流式跟随。
        
- $$$$
    
    **2. 语义压缩：从“嵌套卡片”到“图标+联动视图” (Semantic Compression & Coordinated Multiple Views)**：
    
    - **痛点**：当前把复杂的 Insight 内容直接做成大卡片嵌套在 Storyline 中，不仅浪费空间，还严重破坏了结构的可读性。卡片一小，用户就必须进行极端的放大（Zooming）才能看清。
        
    - **动作**：将所有 Storyline 上的巨型“卡片（Card）”降维，替换为高度抽象的**图标（Icon/Glyph）**形式。
        
    - **动作**：在界面右侧引入**联动视图（Detail View）**。
        
    - **逻辑（学术表述）**：采用宏观-微观设计范式。主视图（Storyline）剥离冗余文本，只负责展示整体的 Pattern（宏观演化态势）；当用户对某个节点感兴趣时，通过“点击”操作，在右侧 Detail View 展示具体的 Insight 内容（微观细节）。
        
- $$$$
    
    **3. 视觉平滑与上下文隐式聚类 (Visual Smoothing & Implicit Contextual Clustering)**：
    
    - **痛点**：当前的线条生硬，且不同分析计划（Plan）之间缺乏视觉上的隔离与联系。
        
    - **动作 - 平滑化**：将 Storyline 的直线/直角连接线全部重构为**平滑曲线（Smooth 线/Bezier Curve）**，使整体布局更加紧凑（Compact）和具有流动感。
        
    - **动作 - 隐式聚类**：对于从属于同一个分析计划（Plan / Session）生成的一系列 Insights，在底层渲染一层**半透明底纹（Background Shading / Hull）**将其包裹或框选起来。
        
    - **逻辑（学术表述）**：无需显式地去画“Plan 和 Plan 之间的关联线”，那会引入过多的视觉杂音（Visual Clutter）。通过底纹的划分，以及节点连接的**密集程度**，用户能够凭借视觉直觉隐式地推断出哪些 Plan 之间的关联更紧密。
        
- $$$$
    
    **4. 数据子集与分析维度的分支表达 (Visualizing Data Subsets via Branching)**：
    
    - **动作**：强化 Storyline 的“分叉（Branching）”语义。
        
    - **逻辑**：Branch 不应仅仅代表生成了新的 Insight，更可以代表 Agent 开辟了新的数据分析子空间（Data Subsets）。例如，模型对数据做了一个 Filter（如“仅关注 2005 年后的数据”），此时就可以在 Storyline 上分支出一条新的主线。这能直观地向用户展示 Agent “关注点转移”的过程。
        
- $$$$
    
    **5. 节点大小的语义映射 (Visual Encoding of Metrics/Importance)**：
    
    - **动作**：完善内部的重要度评估指标（Metrics），并将其直接映射到节点（Icon）的**大小（Size）**上。
        
    - **逻辑**：通过节点大小的错落有致，迅速吸引用户注意力（Motivate）到高价值或高异常的关键节点上。布局算法依然可以调整权重等来平衡 storyline 的基本清晰布局和语义相关性的内容。
        

## 任务分解二：交互设计的系统化 (Interaction Design - Steering)

**目标**：将杂乱的交互点收敛为学术上立得住的“系统化分类”，并脱离单纯的“节点操作”。

- $$$$
    
    **1. 交互分类框架的抽象（重要！）**：
    
    - **动作**：在论文中绝对不能只列举按钮，必须归纳为几个维度的 Steering。例如，可以按操作意图分为 5 个 Dimension，或者按干预粒度进行分类。
        
- $$$$
    
    **2. 解耦 Storyline，引入全局控制（Global Constraint）**：
    
    - **动作**：**不要把所有交互都堆在 Storyline 上！** 增加类似“全局指令列表（Instruction List）”的功能。
        
    - **逻辑**：用户可以随时输入类似“我比较关心 Sales 数据”的 Global Prompt。系统在调度后续子 Agent 时，必须带上这些全局约束。用户可以对这些指令进行增删改。
        
- $$$$
    
    **3. 具体的 Steering 手段实现**：
    
    - **方向引导 (Directional Steering)**：用户可以通过点击图标（Icon），利用 Follow-up questions 开启新的探索分支（Branching 出去）。也可以调整 specific insight 或 specific type（如 correlation）的权重，引导 Agent 深入挖掘。
        
    - **剪枝与阻断 (Pruning/Denying)**：用户若认为某条 Storyline 的分支无意义，可以将其禁用（标灰）。关键点：禁用时，可以让用户输入 Prompt 解释“为什么删掉”，此意图将被 Agent 记忆，防止后续重复分析。
        
- _Prof. Weng 的免责声明_：不要在本作中耗费精力去做“机器自动验证 Insight 正确性（Validation）”的模型，这不是本文重点，真假/好坏交由用户来判断并施加 Steering 即可。
    

## 任务分解三：后端 Agent 架构的重构 (Agent Framework)

**目标**：摆脱现有的“死板工作流（Workflow）”，构建真正的 Long-running Agent。

- $$$$
    
    **1. 从 Workflow 到 Agentic 架构的跃升**：
    
    - **痛点**：Shuhan 犀利指出，现有的架构过于死板，只是一个带有计算图的流程，缺乏 Agent 应有的**自主决策能力（Autonomy）**。
        
    - **动作**：实现一个具有全局视野的**主控节点（Orchestrator / Central Controller）**。
        
- $$$$
    
    **2. Controller 核心逻辑的实现**：
    
    - **动作**：可以通过 Prompt Engineering 简单实现调度逻辑（实在不行可以魔改借用 DeepAnalyze 的底层逻辑）。
        
    - **能力要求**：Controller 能看到全局的分析上下文（Context）；能够自主提出 Plan；能够调用特定的 Sub-agents（Analyzer, Summarizer）；在子 Agent 完成分析后进行汇总（Duplicate check）；**并且能够自主判断何时该暂停分析、抛出中断让用户介入。**
        
    - _注_：不论用户是否初始就给出具体目标（Goal），系统都应能工作。
        

## 任务分解四：相关工作与论文占位 (Related Work & Positioning)

**目标**：理清本文的贡献点，精准回击可能面临的 Reviewer 质疑。

- **论文架构敲定：分为以下三个 Section：**
    
    - $$$$
        
        **2.1 Empowering Data Analysis with LLMs**：
        
        - **逻辑**：梳理从早期的 Turn-based 对话式分析，演进到基于大模型的 Agentic 分析（如 DeepAnalyze 等）。指出当前这类强大方法的缺陷：**缺乏过程的可视化与人的动态介入控制（Unsteerable）**。
            
    - $$$$
        
        **2.2 Visualizing Data Analysis Process**：
        
        - **逻辑**：讨论传统 EDA 过程可视化的工作（如基于 Node-link 的 Data Flow）。指出在 LLM 驱动的复杂非线性场景下，传统可视化不适用，从而引出本文的 Deep Storyline 可视化方案。
            
    - $$$$
        
        **2.3 Interacting with LLMs (取代 Human-LLM Alignment)**：
        
        - **逻辑**：不要自己发明晦涩名词。从 HCI 的角度探讨如何干预大模型。强调本文提供的是一个高度动态、可视化的 Interaction Space。
            
- **如何与高度相似的工作（Light-VA 等）做区隔？ (来自 Shuhan 的核心 Insight)**
    
    - _Light-VA 的本质_：它是一个 Step-by-step 的、模板化的（Templated）需求拆解系统，目的是为了生成特定的 Visual Analytics（VA）系统图表。人介入是为了 Check 当前步骤以进入下一步。
        
    - _本作的本质_：我们面对的是一个 **Long-running 的黑盒自主智能体**。用户的目的不是画图，而是“监控并操纵（Monitor & Control）智能体的发展方向”。场景的复杂度和系统的自主性完全不在一个量级。
        
- **如何与 DeepAnalyze 做区隔？**
    
    - 我们不强调底层模型的 Novelty（甚至可以直接用它的）。我们的核心贡献在于 **Interaction Space 和 Visualization Space** 的设计。
        

## 任务分解五：引言撰写与场景构建 (Intro & Scenario)

- $$$$
    
    **1. 扩写 Introduction**：
    
    - **痛点**：当前两段太短，直接跳入 "To achieve these..." 是不合格的。
        
    - **动作**：严格遵守 Lab 的标准 Intro 模板（4-5段体）。
        
        - Para 1: Background (数据分析的重要性，LLM Agent 在数据分析中的崛起)。
            
        - Para 2: Current Status & Limitation (现有的 Agentic 工具是黑盒，分析路径可能跑偏，用户无法理解其过程)。
            
        - Para 3: Key Challenges (要把这个黑盒过程白盒化，并且允许动态修正，存在哪些交互与可视化上的挑战)。
            
        - Para 4: Proposed Technique (我们的框架如何解决上述挑战)。
            
- $$$$
    
    **2. 构建有说服力的 Scenario (结合 Metrics 和 Interaction)**：
    
    - **动作**：不要只写一个干巴巴的分析结果。Scenario 必须结合我们提出的交互能力（Steering）。
        
    - **剧本设计**：用户在监控过程中，发现系统沿着不感兴趣的路径前进 -> 用户通过干预（Disable branch, add global instruction） -> Agent 调整方向并得到了新的、符合预期的 Insight。即便当前结果不太 Make sense，也必须把这个**动态纠偏的故事（Dynamic Process）**讲圆满。

- **Prof. Weng**: “不要把所有的创新点堆在 Storyline 上。设计该收敛就收敛。不需要为了验证而做检验模型，把选择权交给用户。写 Intro 时要遵循传统 Template，稳扎稳打。”
    
- **Shuhan**: “认清我们的定位：我们不是在做一个传统的 Workflow 工具，而是在给一个会自主乱跑的 Agent 上缰绳（可视化和交互）。一定要区分清楚我们的场景和传统 Step-by-step 系统的本质区别。”
