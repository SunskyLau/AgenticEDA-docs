# 「AI妙记」项目推进详细指南与Action Items

**文档受众:** Yang Tian (及项目Co-authors) **文档目标:** 总结最新会议的所有高价值Comments，明确「数据探索交互框架」的设计方向、Use Case的打磨要求、论文结构写作规范以及评估策略，提供可直接执行的Action Items。 **核心基调:** 不要局限于僵化的层级分类，交互不仅要服务于“干预(Steering)”，也要服务于“理解(Understanding)”；Case的推进必须有极强的动机(Motivation)和领域专家背书。

## 模块一：核心交互框架设计 (Interaction Framework)

**🚨 翁老师的核心定调 (Crucial Insights):**

1. **打破僵化的层级束缚:** 放弃原先“Intent Level x Data/Plan”这种二维相乘的严苛分类。这种分类会极大限制交互类型的拓展。交互动作应该更加独立，基于“用户在EDA过程中想往哪个方向探索”来枚举。
    
2. **交互的双重目的:** 系统的交互不应仅仅是为了“干预(Steering)”Agent的过程，同样重要的是帮助用户“理解(Understanding)”Agent产生的海量Insights和Storyline可视化。
    
3. **Execution Level的简化:** 不需要做复杂的“手动确认模式(Accept/Reject)”。保留基础的**暂停 (Pause)、继续 (Resume)、终止 (Terminate)** 即可。
    

### 📌 具体需要设计与实现的交互动作 (Interaction Verbs)

结合会议讨论，你需要将系统的交互分为以下两大类进行设计和实现：

#### A. 用于干预/引导的交互 (Steering Interactions)

这类交互直接改变Agent下一步的探索方向或行为约束。

- **Focus (聚焦):** * **定义:** 让Agent在未来的探索中聚焦于某个具体的变量（如Sales）或某几个数据单元。
    
    - **效果:** 相当于在Storyline上产生一个分支，后续所有的Insights/Plans都必须与被Focus的变量相关。
        
- **Exclude (排除):**
    
    - **定义:** 明确告知Agent不要关注某个数据范围或探索方向（Focus的反向操作）。
        
- **Clarify (澄清/约束):**
    
    - **定义:** 用户输入额外的指令或领域知识（Domain Knowledge）。
        
    - **操作:** 比如解释某一个Column的特定含义，将其作为全局或局部的约束条件。
        
- **Create / Ask (创建/提问):**
    
    - **定义:** 基于用户的特定灵感（而非现有界面元素）发起新的探索。
        
    - **操作:** 在界面左下角的对话框输入问题（例如：“想知道A和B有没有关系？”）。
        
    - **系统响应:** 系统自动分类用户的Prompt，如果是Question，则自动转化为触发一个新的Plan (Create a plan)；如果是Knowledge，则等同于执行Clarify。
        
- **Elaborate / Dive into (深入探索):**
    
    - **定义:** 针对当前觉得很有意思但探索不够深入的Plan。
        
    - **操作:** 围绕这个已有的Plan，让系统花费更多的Thinking Time，在同一抽象层级生成更多的、额外的Insights（相当于充实当前的Plan分支）。
        

#### B. 用于理解的交互 (Understanding Interactions)

这类交互不改变Agent的运行轨迹，而是帮助分析师消化已经生成的内容。

- **Summarize / Compare (总结/比较):**
    
    - **定义:** 框选或多选几个不同的Plans（例如包含不同变量的探索分支）。
        
    - **操作:** 划拉选择后，调用一次LLM，总结这几个Plan之间的关联和差异（例如：Plan A讲了Gender和Sales，Plan B讲了Region和Sales，总结出它们共同的关键点）。并在右侧Detail Panel显示。
        
- **Trace / Lineage (血缘追溯):**
    
    - **定义:** 帮助用户理解某个特定Insight的来源。
        
    - **操作:** Hover在一个Plan或Insight上，高亮显示所有与它相关联的先决Plans，或者在Storyline上高亮显示与之相关的数据列（Data Columns）的历史演变路径。
        

## 模块二：案例设计与打磨 (Use Case Refinement)

**🚨 翁老师的核心定调 (Crucial Insights):** 当前的Use Case（尤其是妙言设计的Sales数据Case）**动机（Motivation）不足**。分析师做探索的决定不能仅仅因为“某个列在界面上出现频率高”，而必须是因为“发现了有趣的Pattern或强关联”。

### 📌 妙言的Case需要修改的致命点：

- **当前逻辑 (错误):** Global Sales和另外四个列在过滤结果或视图中“出现得比较多/高亮比重大” $\rightarrow$ 用户决定Focus在这几个列上，让系统综合分析。
    
- **修改方向 (正确):** 用户看了初期的几个Summary/Insights $\rightarrow$ 发现这四个列之间可能存在**有价值的业务关联 (Business Correlation)** $\rightarrow$ 产生了一个具体的分析意图 (Analytical Motivation) $\rightarrow$ 使用 **Create/Summarize** 交互，让Agent专门针对这四个变量创建一个新的综合分析Plan。
    

### 📌 设计Case的黄金法则 (4-5步法)：

你和妙言在设计新的Case时，必须严格按照以下模板写出4-5步：

1. **用户看到了什么** (e.g., 在Storyline分叉处看到了关于Sales和Region的两个矛盾的Insight)。
    
2. **产生了什么业务思考/动机** (e.g., 怀疑是不同厂商营销策略导致)。
    
3. **执行了什么交互动作** (e.g., 框选这两个Insight，执行 `Summarize`，然后通过对话框 `Ask/Create` 一个关于营销策略的Plan)。
    
4. **系统产生了什么视觉反馈/效果** (e.g., 生成了一个新的Plan Node，并输出了隐藏在表面下的Risk Report)。
    

## 模块三：论文结构与写作规划 (Paper Structure & Writing)

根据翁老师和舒涵的建议，论文结构需要做如下调整：

### 1. Section 3: Background & Problem Formulation (取代传统的Motivating Case)

- **不要用普通大模型（如ChatGPT）去硬写一个“对比Case”。** 这样很难写，因为传统大模型本来就没有固定的EDA workflow。
    
- **正确写法:** 写成 `Problem Formulation`。
    
    - **背景:** 介绍长程Agent (Long-running Agent) 在执行数据探索(EDA)时的场景。
        
    - **挑战:** 阐述在这种长程自主运行的环境下，人类专家与Agent协作所面临的核心难题（例如：Agent容易跑偏、人类难以理解Agent生成的海量中间结果、缺乏细粒度的引导手段等）。
        

### 2. Requirements / Design Goals (舒涵建议)

- 如果不采用 `Tasks` 的写法，可以写成 `Design Goals` 或 `Requirements`。
    
- 围绕系统最终能呈现的功能和上述推导出的Problem来反向设定。
    

### 3. Usage Scenario (系统演示场景)

- **位置:** 放在 Requirements 之后，System 介绍之前（或者融合在 System 章节中，讲到哪里写到哪里）。
    
- **目的:** 充当“系统说明书”。用一个**独立于Case Study的、简单的数据集**。
    
- **内容:** 尽可能覆盖系统**所有的核心功能和交互**（如何读视图、如何做Focus/Summarize等），不需要很深的领域洞察，重点在于“Showcase System Capabilities”。
    

### 4. System Section (系统章节)

- 建议分为两个子章节来写：
    
    - **Visual Understanding:** 介绍可视化视图（Storyline等）以及配合视图的**理解型交互**（Trace, Summarize）。
        
    - **Interactive Steering:** 专门介绍提出的**干预型交互框架**（Focus, Exclude, Clarify, Create等），并配上小图讲解每个Interaction的设计Rationale。
        
- **注意:** 不要孤立地论证某一个交互动作的Contribution。Contribution在于整套包含“可视化+引导+理解”的 **Cohesive Framework**。
    

## 模块四：评估策略与实验规划 (Evaluation Strategy)

为了让论文有说服力，必须进行严谨的评估。翁老师明确要求了以下组合：

### 1. Case Study (基于真实领域的专家案例)

- **核心要求:** **必须找对应领域的专家来背书！** * **操作方式:** * 选定数据集（例如：如果是体育数据，去找体育组的老师/同学；如果是经济数据，找商学院/经济学背景的研究生）。
    
    - 不要自己捏造Insights，因为你无法判断“把这四个列放在一起分析到底合不合理”。
        
    - 让专家带着他们真实的探索需求，使用你的系统跑完整个EDA过程（结合专家访谈 Expert Interview 一起做）。
        
- **数量:** 建议2个不同的Case Study（配合2个不同的数据集，且不要与Scenario使用的数据集重合。总共需要3个数据集）。
    

### 2. User Study (可选但强烈建议：受控的可用性测试)

- **形式:** Task-based controlled study。
    
- **操作方式:** 预设好一条EDA的路径和一系列具体的Task List（例如：“请找出导致XX下降的核心原因”、“请利用系统阻止Agent继续分析XX”）。
    
- **目的:** 让普通用户执行这些任务，收集可用性评分（Usability Score，如SUS），主要用于证明系统是易用的、好上手的。
