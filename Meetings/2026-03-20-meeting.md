## 模块二：系统设计与交互优化 (System & Steerability)

_这是本次讨论的重中之重，当前系统的 Steerability 较弱，必须进行根本性的交互逻辑替换。_

### 1. 废弃“右键菜单”与“行内编辑 (inline-edit)”

- **Insight (Shuhan)：** 交互设计的核心前提是 **Discoverability (可发现性)**。右键菜单非常反直觉，用户根本不知道哪里可以点。仿写 Copilot 的行内文字输入逻辑过于复杂且没有必要。
    

### 2. 重构“明确的工具栏交互 (Explicit Tool Interactions)”

- **任务：** 界面上提供一排固定/明确的工具栏（如三支笔：Focus笔、Ignore笔、Elaborate笔）。
    
- **交互工作流 (Workflow) 必须按以下设计：**
    
    1. **激活工具：** 用户点击工具栏的 `Ignore` 笔。
        
    2. **视觉提示 (Bounding Box)：** 用户的鼠标移到画面上时，任何支持 Ignore 的元素（如 Insight 卡片、Summary 文本）都会出现一个 Bounding Box（提示框/高亮边框），明确告诉用户“这个东西是可以操作的”。
        
    3. **触发操作：** 用户点击特定的 Insight 卡片或文本。
        
    4. **后端询问 (Ask Clarifying Question)：** 前端将该段文本传给后端 Agent，Agent 从中提取 3-4 个关键特征/实体（例如：年份 2008、峰值趋势、某个特定类型）。
        
    5. **用户二次确认：** 界面旁边弹出一个小的 Tips/输入框，列出这 3-4 个选项让用户进行**单选或多选**（明确到底 Ignore 什么细节）。如果全都不对，提供一个 `Other` 输入框让用户自己打字（参考 Claude 的做法）。
        
- **技术边界：** 不需要把整个界面的所有上下文都传给大模型做复杂的理解，**只传用户用笔选中的那一段特定文本（Insight 或 Summary）**。
    

### 3. 操作粒度与对象 (Granularity)

- **Column 级别：** 支持多选（Ctrl+点击），可以同时选择多个 Column 告诉后端 Ignore / Focus。
    
- **Insight / Summary 级别：** 只需要支持**单选**。人类的认知一次只能处理一个核心 Insight，不需要做多个 Insight 的组合控制。
    
- **Plan / 宏观级别：** 原先放在 Insight 层级的 Plan 操作（比如新建/修改分析逻辑），**必须下移到 Execution Level**。因为对 Plan 的操作不属于画面上的数据实体分析，而是底层执行逻辑。
    

### 4. 状态可视化完善 (State Transparency)

- **任务：** 当一个 Plan 正在创建、还未生成具体的 Summary 或图形时，**主视图（Storyline/画布）上必须有占位符 (Placeholder) 和加载动画 (Loading/Spinner)**。不能只在侧边栏显示“Agent正在运行”。主视图必须响应所有的状态。

## 模块四：评估与实验设计 (Evaluation)

_采用 "User Study 排雷 + Case Study 冲刺" 的双轨策略。_

### 1. 简易的用户体验测试 (Pilot/User Study)

- **目的：** 验证系统鲁棒性，获取 SUS (System Usability Scale) 评分，作为量化指标。
    
- **招募：** CC98 发帖，招募 12 名本科生（给报酬）。
    
- **流程：** 每人 40 分钟以内。给定一个所有人都看得懂的简单数据集（如：学生成绩数据）。给定一些具体的 Task（确保他们能 cover 到 Focus, Ignore, Plan 等功能），做完后填 SUS 问卷表。
    
- **访谈要求：** 不需要做长篇定性访谈。只针对“打低分”的人问为什么（找系统的茬），针对“高分”问喜欢什么。
    
- **前置任务：** 在正式上线前，找身边的熟人（如女朋友）做一个 Pilot Study，观察他们会不会搞出“幺蛾子”，确保系统足够鲁棒。
    

### 2. 专家案例研究 (Expert Case Study)

- **数据集：** 1. 任天堂游戏销量数据；2. 体育数据集（Zihao 提供）。
    
- **模式：自由探索 (Free Exploration)。** 既然妙言确认了系统的结论具有鲁棒性，就让专家（如 Zihao 或具有金融/经济背景的人士）自由使用，这是比 "Use Case" 更强有力的学术证据。
    
- **策略分布：** 两个 Case 要体现**功能互补**。写作时详略得当。比如，可以在向专家 Warm up（热身演示）时，通过提示词模板暗示他们：在 Case 1 中多使用 Focus/Ignore 控制细节，在 Case 2 中多尝试宏观的 Plan/Elaborate。
    
- **[核心技术要求] 录制与日志保存：**
    
    - 必须在后端留一个接口：**将前后端交互的 Log 存为 JSON 文件。**
        
    - **原因：** Case Study 会非常长。在后期剪辑 Demo 视频时，不需要痛苦地从长视频里剪，只需要拿保存下来的 JSON Log，在系统里一键复现 (Replicate) 整个交互流程，然后再录制干净的 Demo 视频（关键步骤原速，跑大模型的过程 10x 加速）。

### 3. Design Goals (第三章 - 需要重写，极其关键)

_原先的 Design Goals 粒度混乱，将 G6 这种代表整个系统的宏观目标与细碎的目标并列，这是大忌。请按以下 **3+2 结构** 重组：_

**针对 Transparency (理解维度)：**

- **G1: Visualize Data Artifacts at Different Levels of Granularity.** (支持不同粒度数据元素的可视化，如 Column -> Insight -> Plan)。
    
- **G2: Unveil the Evolvable Relationships and Reasoning Logic among Elements.** (揭示元素之间的演进关系与 Agent 的推理逻辑，这直接导向了 Storyline 的设计)。
    
- **G3: Support Intuitive Comparison across Parallel / Multiple Decision Branches.** (支持对多分支、并行分析结果的直观比较，体现 Agentic 区别于线性对话的优势)。
    

**针对 Steerability (操控维度)：**

- **G4: Enable Granular Control over Reasoning Artifacts.** (提供针对不同粒度细碎元素的干预能力，如基于 Bounding Box 的 Focus/Ignore)。
    
- **G5: Support High-level Steering on Decision Making (Execution Level).** (支持在执行层面/高维度的方向干预，如控制 Plan 的走向和 Branch)。
    

_(注：Problem Formulation 的设定，直接说“这些是基于 Expert Interview 总结出来的”即可，没有人会质疑这部分，属于合理的开篇设定。)_

### 4. System / Framework 章节结构

_避免 Transparency 和 Steerability 章节之间出现大量内容重复。_

- **第一小节：** 简述 Framework 和 Agent 机制。
    
- **第二小节 (Understanding/Transparency)：** 围绕主视图写。先写各层级的 Visualization Elements（对应 G1），再写 Storyline 的宏观布局与树状关系（对应 G2, G3）。
    
- **第三小节 (Steering/Interaction)：** 针对上述的每一类 Element，具体介绍我们设计了什么工具（三支笔、Ask Clarifying Question），以及对应的效果（对应 G4, G5）。
    

## 💡 Shuhan 的核心学术洞察与叮嘱 (For 杨天 & 团队)

1. **“不完美但必须有”：** Steerability 的交互（比如 Ignore/Focus Insight 后弹出的明确选项）实现起来可能比较复杂，你们可以做得简陋一点（比如让大模型抓词可能会出错），但**系统里绝对不能没有**。如果没有这个机制，系统就沦为单纯的“Help user understanding”，彻底失去了“Steerability”的 Claim，这篇论文就垮了。
    
2. **“交互的不可见性是反直觉的”：** 不要让用户去试探系统（比如右键菜单），要把能力明确地放在台面上（工具栏 + Bounding Box），这是解决人机协作的核心 HCI 原则。
    
3. **“要有信心”：** Shuhan 明确表示她非常喜欢这个 Idea，认为它具有很强的前瞻性（Pro-looking/Promising）。现有的 Data Validation/Verification 已经做烂了，但 Long-running Agent 的可视化交互才刚刚起步。如果做好，后续肯定有人 Follow。
    
4. **“力度的统一”：** 无论是在写 Design Goals 还是设计系统功能时，一定要注意“粒度 (Granularity)”的统一。不要把宏观的架构设定与一个微小的 UI 标签并列讨论。
