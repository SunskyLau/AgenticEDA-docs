## 核心洞察 (Golden Insights from Co-author)

_这些 Insight 是支撑我们论文动机（Motivation）和贡献（Contribution）的核心基石，在写作时必须深刻理解并反复强调。_

1. **真正的 EDA vs. 传统 VA (Visual Analytics)**
    
    - **传统 VA (如 LightVA)**: 是高度模板化 (Templated) 的。往往拥有一个明确的、初始的全局目标 (Single goal/tree)，数据定期更新，分析流程是固定的复用过程。
        
    - **我们的 Agentic-EDA**: 强调**高度的动态性 (Dynamic)**。在 EDA 初期，用户可能没有任何明确目标（例如：“帮我看看这批数据有什么意思”）。系统需要在广度和深度上同时探索，用户在 Agent 运行的中间过程 (Intermediate process) 中会不断产生新的 Insight 并做出新的决策。这是 Long-running 场景独有的价值。
        
2. **Multi-step vs. Single-step 的本质区别**
    
    - **Single-step (如早期工具)**: 端到端 (End-to-end)。无论丢给系统什么任务，它一定会一股脑跑完给出一个 Output。用户只能在跑完后，把结果作为上下文开启下一轮 (Turn-by-turn)。
        
    - **Multi-step (我们的场景)**: Agent 具有自主中断 (Pause/Interrupt) 的能力或机制。它在运行状态机时，会在中途发现问题或需要确认时停下来。这种“长程运行”决定了我们必须提供细粒度的干预能力。
        
3. **直接可视化交互 (Direct Steering) vs. 对话交互 (Chat-based)**
    
    - 为什么不在 Chat 里直接说“往深了分析”？因为**自然语言具有极强的模糊性**。基于 Chat 的交互本质上依然是“回合制”的间接意图表达。
        
    - 我们将交互直接绑定在可视化图表（Storyline / Graph）上，是因为图表提供了一个**精准无歧义的操作空间 (Space)**。这赋予了人类更高的 Agency，是我们最核心的 Contribution。

### 2. 对比与 Baseline 定位 (如何确立护城河)

在描述 Limitation 或 Related Work 时，按以下口径对比：

- **对比现有的 Data Analysis 工具 (Wait GPT, Insight Lens 等)**:
    
    - 它们是 Single-step / Turn-by-turn 的交互。
        
    - 受限于此，它们只能做线性的 (Linear) 可视化，强迫用户陷入底层的 Execution flow (执行流) 中。
        
    - **我们的优势**: 我们能做 **Across plan** (跨计划/跨分支) 的比较，支持并行分析，真正拔高到 High-level sensemaking。
        
- **对比纯大模型 (Claude Code, Codex, Cursor)**:
    
    - 即便是这些最先进的模型，它们目前的交互范式依旧是 **Conversation (文本对话)**，依然是一轮一轮的回合制。人类只能通过间接的 Prompt 调整目标，而不能显式地 (Explicitly) 在运行中控制下一步 (Next action)。
        
    - **我们的优势**: 提供 Direct / Explicit 的交互机制，直接干涉 Agent。

## 二、 交互设计空间精简与重构 (Design Space Refinement)

_核心痛点：之前的交互词汇 (Focus, Suppress, Reframe, Inform, Control) 过于抽象 (High-level) 且存在语义重叠，导致设计混乱。必须回归到具体的可视化对象 (Objects) 上。_

### 1. 明确交互对象 (Objects on Storyline)

我们的 Storyline 视图上只有三种核心实体可以被操作：

1. **Plan (计划)**: 宏观的分析路线。
    
2. **Insight (洞察)**: 具体的分析发现。
    
3. **Path / Column (路径/数据列)**: 具体的执行线索。
    

### 2. 重构交互原语 (Interaction Primitives) - 极简“4+1”模型

抛弃复杂的分类，直接在上述对象上提供最原子化、最明确的操作。分为两大类：**意图干涉 (Intent)** 和 **执行干涉 (Execution)**。

#### A. 意图调整 (Intent) - 针对宏观方向

_这部分旨在告诉模型“接下来该怎么想”。_

1. **深度 (Depth / Dive into)**: _(Single Object)_ 拖拽或点击某个节点，让 Agent **围绕该点继续深入分析**。
    
2. **广度 (Breadth / Branch out)**: _(Single Object)_ 点击分叉图标，让 Agent **围绕该点进行多角度的横向发散分析**。
    
3. **剪枝/拒绝 (Cut-off / Discard)**: _(Single Object)_ **彻底 Ban 掉**某个方向。明确告诉系统“不再分析这个相关内容”（例如：对质量分析不感兴趣）。
    
    - _UI 修正细节_: 必须与单纯的“停止执行”区分开。不要让用户猜。可以在线上直接加一个外置的 Chip 或悬浮气泡 (Pop out panel)，文字写明：“不再分析质量？”，点击打叉代表方向上的彻底抛弃。不要用模糊的 "Suppress/降权"。
        
4. **建立关系 (Relation)**: _(Multi Object)_ 允许用户多选几个 Plan 或 Insight，让 Agent 分析它们之间的联系。
    

#### B. 执行干预 (Execution) - 针对底层运行

_这部分旨在告诉模型“这个动作停下来”。_

1. **终止/取消 (Terminate)**: 直接点叉，立刻停止当前 Plan/Path 的运行。不涉及后续意图的改变，仅仅是“掐断”当前的计算。
    

### 3. 必须砍掉的伪需求 (减负，不要实现)

- **插入计划 (Insert Plan)**: 砍掉。在系统高速运行时人工插入计划极易导致状态崩溃（如 Codex 跑一半插入会生出一堆乱七八糟的 sheet）。真实场景下，用户更倾向于先暂停再规划。
    
- **补充信息 (Inform / Chat-based fallback)**: 扔回全局 Chat 框里，作为兜底选项即可。这不是我们论文的核心贡献（大家都有的功能，改 System prompt 毫无新意）。**与 Storyline 可视化无关的交互全部从核心 Contribution 里剔除。**
