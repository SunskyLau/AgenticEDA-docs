## 模块一：UI 界面与交互体验优化 (UI/UX Refinement)

此模块的核心目标是：**保持中间画布区域的清爽与对称，降低用户的认知负荷，实现跨视图的丝滑联动。**

### 1. 布局与视觉降噪

- [ ] **缩小工具调用 (Tool Call) 卡片:** 左侧页面展示的“工具调用”卡片（如代码执行过程等）尺寸过大。需要将其高度和整体体积压缩，保持界面清爽。
    
- [ ] **优化 Plan 面板显示:** 当生成新的 Plan 时，需要在 Plan 面板中完整显示所有的子任务（Sub-tasks），便于用户全局把握。
    
- [ ] **界面元素对称与画布留白:** - 将 Plan 面板和 Insight 标签页分别移动至界面的**左上角**和**右上角**，形成视觉对称。
    
    - **核心原则:** 必须保证中间的“画布（Canvas）”区域没有任何遮挡，留出足够的空间用于后续交互。
        

### 2. 多视图联动 (Linked Views)

- [ ] **实现 Insight 与各面板的双向跳转:**
    
    - 中间的 Insight 卡片必须是**可点击的 (Clickable)**。
        
    - 点击某个 Insight 时，系统自动高亮或跳转到左侧的 Summary 区域。
        
    - 同时，右侧面板也应联动跳转到对应的数据/图表位置，实现“Summary - Insight - Detail”的完美联动。
        

### 3. Steering (用户干预/引导) 交互设计

- [ ] **Focus 悬浮面板交互优化:**
    
    - **支持拖拽 (Draggable):** 当用户触发 Focus、Elaborate 等 Steering 操作弹出卡片时，在卡片右上角增加拖拽把手（Drag Handle），允许用户将其移开，以免遮挡底层信息。
        
    - **取消 Ctrl 键限制:** 用户选择多个特征或列时，不需要按住 `Ctrl` 键，直接连续点击即可复选，再次点击取消。
        
    - **状态互斥与置灰 (Gray-out):** (可选/视开发时间而定) 当用户选中某个 Column 时，其他不可选的元素应自动置灰，明确提示当前的可选范围。
        
- [ ] **执行队列调度 (Priority Queue / Concurrency Control):**
    
    - **暴露并发参数:** 允许用户在前端设置 Agent 的线程并发数量（默认 2，最大可调至 3 或 5）。
        
    - **拖拽调整优先级 (Drag-to-Swap):** 系统同时生成了 5 个 Plan，但由于并发限制只有前 2 个在跑（Running），后 3 个处于等待（Pending）。
        
    - **交互逻辑:** 用户可以拖拽下方的 Plan 到上方。这里的逻辑**不是链表插入（挤上去）**，而是**交换 (Swap)**。用户将正在跑的 Plan 暂停或拖下来，与目标 Plan 交换位置，以此实现人为介入的资源调度。
        

## 模块二：Prompt 重构与 Agent 逻辑优化

此模块的核心目标是：**分离 System Prompt 与 User Prompt，向用户展示极简、确定的意图，将复杂的约束下放至后端。**

### 1. 极简的用户端 Prompt 呈现

- [ ] **精简确认弹窗 (Confirm Modal) 中的 Prompt:** - 目前发送给大模型的 Prompt 太过复杂（包含了大量 background、other keywords 等前端状态）。
    
    - **改进:** 用户端看到的 Prompt 必须非常直接、清晰（例如：`"I hope to focus on sales distribution and outliers."`）。
        
    - **交互:** 用户可以点击生成这个默认的简洁 Prompt，如果不满意，可以直接在输入框内**原地修改 (In-place edit)**，确认无误后发送。
        
- [ ] **处理 Elaborate / Ignore / Focus 的指令:** - 核心只保留触发动作的第一句话。完全没有必要在给用户的 Prompt 预览中附带冗长的上下文。
    

### 2. 前后端 Prompt 职责分离 (System vs. User Prompt)

- [ ] **System Prompt 负责约束:** 将原先写在 User Prompt 里的复杂规则（如“你在进行 insight 深度分析时不能做某事”、“格式要求”等）全部转移到 **System Prompt** 的 `instruction` 部分。
    
- [ ] **User Prompt 负责状态/时间线:** User Prompt 仅需传递当前的时间线状态（Timeline State）和用户的具体指令类型（打上 Label，如 `Type: Elaborate`, `Target: keep investigation...`）。
    
- [ ] **Prompt Debugging 任务:** 在后台打印或存储几次完整的 System Prompt 和 User Prompt。人工审查，剔除“废话”和不符合预期的指令，确保 Agent 行为受控。
    

### 3. Agent 的主动反馈机制 (Respond to User)

- [ ] **增加 "Evaluate Progress & Respond" 环节:** - 目前 Agent 只有在用户主动 Steering 时才会回复。这是不自然的。
    
    - **改进:** 在一轮分析（多个 Plan）完成并进行 Stage Summary (Evaluate Progress) 之后，Agent **必须主动调用 `respond_to_user` 工具**。
        
    - **目的:** 用自然语言向用户解释：“我刚才分析了什么，我接下来的高级意图/规划是什么”。后续生成的 Plan 必须围绕这个“高级意图”展开，让用户感知到 LLM 的思考逻辑。
        

## 模块三：系统鲁棒性与 Bug 修复

### 1. 解决代码执行上下文丢失问题 (NameError)

- [ ] **修复 Python 运行环境的 State 丢失:**
    
    - **现象:** Agent 生成的代码经常报错 `NameError`。
        
    - **原因定位:** 大模型将两段代码视为连续的，引用了上一段代码的变量；但我们的系统环境在执行新代码段时，丢失了之前的上下文（相当于变量被清空了）。
        
    - **解决方案:** 必须让代码执行环境保持会话连续性（类似于 **Jupyter Notebook / Jupyter Hub** 的机制），保留上一轮运行的全局变量。如果无法接入成熟的沙箱环境，必须在系统内部修复环境持久化逻辑。
        

### 2. 数据集解析 Bug 修复

- [ ] **修复 Filter 的列名解析:** 右上角的 Filter 无法正常识别列，原因在于当前数据集解析可能强依赖于逗号（`,`）分隔符。需要修复解析逻辑，处理缺失列名或分隔符异常的情况。
    

## 模块四：用户研究 (User Study) 实验设计

此模块的核心目标是：**通过强引导式的任务，确保被试能够体验到系统的所有核心功能（特别是 Steering 和 Insight 发现），为 SUS 打分和定性访谈提供依据。**

### 1. 数据集准备 (需准备两份)

- [ ] **数据集 A：Warm-up (热身/破冰数据集)**
    
    - 特点：简单、直观、无需太强领域知识。
        
    - 目的：在正式实验前，用大约 5 分钟向用户讲解系统全局功能，并让其上手随意操作，熟悉界面。
        
- [ ] **数据集 B：Task (正式实验数据集)**
    
    - 来源：已确定的“学生数据集”。
        
    - 处理：删除冗余列，保留用户易于理解的核心列即可。必须确保该数据集在当前系统下能分析出**有意义的 Insights**，否则系统会被评价为“鸡肋”。
        

### 2. 实验任务设计原则 (Guided & Task-oriented)

- **绝对不要给开放式的分析目标**（如“请你找出十个有趣的结论”），这会导致不可控。
    
- **必须采用保姆式的指令 (Step-by-step Execution):** 通过研究员的指令，强制用户遍历所有核心 Feature。
    
- **实验剧本 (Script) 示例:**
    
    1. _“系统已完成初始分析，请你阅读一下目前生成的这个 Insight，理解它在说什么。”_
        
    2. _“现在，我希望你指挥 Agent 进一步深度探索这个 Insight。请你使用 Focus/Elaborate 功能（触发 Steering）。”_
        
    3. _“请你拖拽优先级面板，将某个处于 Pending 的任务提前。”_
        
    4. _“请你利用右上角的 Filter，筛选出所有包含 `[特定属性/关键词]` 的 Insight，并告诉我总共有几个。”_
        
- **目的:** 被试在完成这一套动作后，切实体会到了“人机协同分析”、“干预 Agent”的作用。这为后续填写 **SUS（系统可用性量表）** 以及**半结构化访谈**提供了扎实的体验基础。
