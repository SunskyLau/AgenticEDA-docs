# AI驱动数据分析系统与可视化优化执行指南 (Action Plan)

**文档受众:** Yangtian (First Author / Project Lead) **会议核心参与者:** Yangtian, Miaoyan (Dev), Shuhan (Storyline Expert), Prof. Weng (Advisor) **项目目标:** 完善基于多智能体(Multi-Agent)的数据分析系统，优化Storyline可视化布局以支持用户引导(Steering)，并完成论文第三章(Formative Study/Problem Formulation)的撰写。

## 核心指导思想 (Executive Summary)

1. **摒弃重新发明算法：** 不要试图去创新Storyline的底层布局算法（如跨线最小化等），而是复用现有的成熟算法（参考iStoryline, StoryFlow），将核心贡献放在**“可视化Pattern与分析语义的映射”**以及**“支持大模型动态分析的Progressive（渐进式）布局”**上。
    
2. **强调分析语义的视觉化：** Storyline中的线、分支、断点、聚合，必须能直接反映AI Agent的分析逻辑（如假设提出、验证分支、结论合并、变量相关性消失等）。
    
3. **写作逻辑重于定式：** 论文第三章（Problem Formulation / Design Requirements）需要通过一个强有力的“Motivating Scenario（动机场景）”来引出传统流程的痛点（Challenges），并逻辑严密地推导出系统的设计需求（Requirements）。
    

## 模块一：Storyline 可视化核心优化 (由 Shuhan & Prof. Weng 指导)

当前Storyline存在线条剧烈波动、交叉过多、视觉混乱且缺乏逻辑表征的问题。必须按照以下策略进行彻底重构：

### 1.1 视觉模式与分析语义的映射 (Visual Patterns to Semantics)

_关键洞察(Shuhan): Storyline不仅是画线，必须让用户一眼看出AI在干什么（Pattern Design Space）。_

- **分支 (Branching / Splitting):** * **语义:** 代表分析方向的分歧或多重假设验证。例如，发现“销量下降”，Agent可能会分出两个Plan：一条线去分析“季节(Seasonality)”，另一条线去分析“价格(Price)”。
    
    - **视觉表现:** 一根代表目标指标的线，在特定时间点分叉为两条，分别与不同的Column（数据列）产生交互。
        
- **合并 (Merging / Linking):**
    
    - **语义:** 代表综合多个证据得出最终结论（Conclusion / Synthesis）。例如，上述两个方向分析完毕，发现都相关，最终线汇聚在一起产生一个综合的Insight。
        
- **出现与消失 (Appearance & Disappearance):**
    
    - **语义:** 代表某个数据列(Column)在当前分析阶段的“相关性”或“重要性”。
        
    - **视觉表现:** 当某个变量不再相关时，不要强行让它继续画线并产生扭曲。直接**打断（Cut-off）**或**极度淡化（变成极细、几乎透明的线）**。只保留当前Session/Plan中真正用到的数据列。
        
- **跳跃/重现 (Jumping / Re-appearance):**
    
    - **语义:** 某个先前被忽略的变量，在后续的深度分析或最终Report总结中再次变得重要。
        
    - **视觉表现:** 线条断开后，在后续的X轴节点重新出现。
        

### 1.2 渐进式布局与稳定性约束 (Progressive Layout & Stability)

_关键洞察(Prof. Weng): 系统是动态生成的，布局不能在每次新生成时发生全局剧变（Perception跳变），这会导致用户认知崩溃。_

- **锁定历史布局 (Lock Previous Layouts):** 当Agent生成第二轮（Round 2）的分析结果时，**严禁**为了全局交叉最少而回退去大幅修改第一轮（Round 1）的线条Y轴顺序和节点位置。
    
- **局部优化 (Local Optimization):** 只能在当前新生成的一轮（例如Round 2内部）进行顺序优化（Best effort减少交叉），然后将其接到已锁定的Round 1末尾。
    
- **轮次视觉分割:** 在视觉上明确区分不同的分析轮次。例如，在Round 1和Round 2之间使用**灰色虚线**进行垂直对齐和阶段分割，甚至可以让数据线在进入下一轮前做一次轻微的“向中间汇聚对齐（类似平行坐标系节点的聚拢）”，再重新Branch出去，以增加美观度。
    

### 1.3 视图层级与缩放设计 (Level of Details - LOD)

- **固定尺寸元素:** 图表中的Icon、Label或文本卡片，**不要**随着地图的放大（Zoom In）而无限等比放大（无信息量的放大是无用的）。应该参考高德地图的POI设计，维持固定尺寸。
    
- **语义缩放 (Semantic Zooming):** * 在缩小（Zoom Out / Overview）级别，只显示小节点、颜色底纹和1-2个关键词（Keywords）。
    
    - 在放大（Zoom In / Detail）级别，节点展开为详细的信息卡片（卡片尺寸本身不随Zoom无极放大，而是呈现更多文本）。
        

## 模块二：UI 界面与交互系统完善 (由 Prof. Weng 指导)

为了支持上述可视化，系统UI需要进行结构性调整。摒弃早期纯文本Topic堆叠的设计。

### 2.1 分析会话的分组与呈现 (Session & Plan Grouping)

- **背景底纹 (Background Grouping):** 同一个Plan（或Agent Session）生成的Insights，在Storyline上应当用同一个**底色色块（Area/Background）**框起来，明确表示这是一个场景(Scene)或一次独立的探索方向。
    
- **平行执行 (Parallel Execution):** 不同的Plan在逻辑上如果是并行的，不需要强行在时间轴（X轴）上串行排布。可以在垂直方向（Y轴）分成上、中、下三个隔离的区域（Session段），数据流在进入这些并行Session前先进行Branch（分流）。
    

### 2.2 辅助视图：状态监控与全局过滤 (Auxiliary Views)

- **顶部状态栏 (Agent Status View):**
    
    - **功能:** 必须有一个明显的区域显示当前“正在运行的Agent任务”。
        
    - **设计:** 显示如 "2 Agents Running..."。展开后，有一句话描述（One-sentence summary）每个Agent的具体Plan。点击该状态，可以联动高亮Storyline中对应的正在生成的区域。
        
- **侧边/底部列表视图 (Insight List & Filter View -** _**非常重要**_**):**
    
    - **功能:** 弥补Storyline在精确检索上的不足，提供所有Insights的“目录式”详情视图。
        
    - **交互设计:**
        
        - 提供一个扁平的List View（列表），可以按照Importance（重要性）进行排序（Sort）。
            
        - **Coverage Map / Histogram:** 将此前的矩阵图简化。做一个小型的柱状图（Bar Chart / Stacked Bar Chart）。X轴是不同的Data Column，Y轴是Insight的数量，堆叠颜色表示Insight Type。
            
        - **联动刷选 (Brushing & Linking):** 点击这个直方图的某个Bar（某个Column），Storyline和List View同时过滤或高亮，只显示与该变量相关的Insights。
            

### 2.3 用户引导交互 (Steering Interactions)

- **交互删减:** 针对不满意的探索方向，除了现有的Chat对话打断外，支持用户在界面上“直接点击某个Plan区域，直接删除/停止”。
    
- **视觉联动:** 悬浮(Hover)在List View的某个Insight上时，Storyline中对应的那条路径需要被高亮（Highlight Path），展示它经过了哪些Session。
    

## 模块三：论文撰写策略 (Section 3: Formative Study)

这是本次会议讨论的重中之重。请严格按照Prof. Weng的指导重构论文的Section 3。

### 3.1 核心写作逻辑架构 (Structure & Flow)

无需拘泥于固定的八股文（如必须写Data Description），但必须有极强的**逻辑递进**。推荐采用以下结构：

1. **Motivating Scenario (动机场景):**
    
    - **内容:** 构思一个完整的故事。描述在传统的、基于对话的大模型数据分析工具（如Advanced Data Analysis）中，用户是如何工作的。
        
    - **痛点暴露:** 用户输入一个Prompt -> 模型进行漫长的黑盒思考(Thinking) -> 吐出一大堆文字Results -> 用户极难阅读和理解这些零散的Insights内部逻辑 -> 用户需要在未充分理解的情况下，硬着头皮输入下一步指令（Menu驱动或盲目Follow up）。
        
2. **Formulating Challenges (提炼挑战 / C1, C2...):**
    
    - 从上述Scenario中，明确提取出当前流程的痛点（Pain points）。例如：
        
        - _C1: Loss of Global Analytical Context (丢失全局分析上下文，难以理解模型思考路径)。_
            
        - _C2: Difficulty in Intervening/Steering Mid-process (难以在黑盒思考过程中进行精细化干预)。_
            
    - _⚠️注意区别: 这里的Challenge是指“用户使用传统工具的痛点”，**绝对不能**和Introduction部分提到的“技术实现难点（Technical Challenges，例如算法性能、大模型幻觉等）”混淆。即使都要列C1/C2，所指代的内容完全不同。_
        
3. **Design Requirements / Considerations (设计需求 / R1, R2...):**
    
    - 基于C1, C2，逻辑严密地推导出我们的系统需要具备什么。
        
    - 必须体现出“孵化（Evolve）”的过程：因为有C1，所以我们需要R1 (提供具备分析语义的全局可视化Timeline)；因为有C2，所以我们需要R2 (支持Progressive的流式展示和即时的交互打断/引导)。
        

### 3.2 专家访谈执行计划 (Expert Interviews)

- 为了支撑Section 3的合理性，不能闭门造车，需要进行Expert Review式的讨论总结。
    
- **寻找对象:** 找至少3-5位在**大模型应用或数据分析**方面有经验的Senior人员。
    
    - 内部推荐：Zikun、体育数据分析组的同学等。
        
- **访谈方式:** 1. 先基于本文档将系统设计图（Figma/原型）修改完善。 2. 拿着修改后的设计稿去找专家聊，抛出Motivating Scenario。 3. 收集反馈（哪怕是确认了你的痛点确实存在），并在论文中写明：“通过与领域专家的合作访谈，我们总结并确认了以下Design Requirements...”
