# 会议记录 - 2026-01-27

## 一、Insights Map Layout 设计方案

### 1.1 Layout 方法论

**三种主要方法**

1. **纯语义相似度方法**
   - 完全基于语义相似度（semantic similarity）进行布局
   - 适用于 insights 之间关联性强的场景

2. **复合 Metrics 方法**
   - 按 insight 类型分类，将平面划分成不同区域
   - 先将 insight 放入对应区域，再在区域内进一步细化 layout
   - 结合多维度信息进行布局优化

3. **基于数据探索的方法**
   - 按照探索的数据列（column）进行区域划分
   - 例如：某区域的 insight 与某个 column 相关，另一区域与其他 column 相关
   - 数据范围也可作为划分依据

**实施策略**
1. 先运行多个实际 case，了解整体情况
2. 根据 case 的性质判断哪种 layout 方法更合适
3. 设计多种 layout 方法，用实际 case 进行测试和对比
4. 综合考虑多种方法，提出一个综合的最佳设计方案

### 1.2 复合 Metrics 计算方法

**主要 Metrics**
- Insight 类型
- Insight 之间的相似程度
- Insight 所涉及的数据范围

**计算策略**
- **分层计算**：先考虑某一 metric，再考虑其他 metric
- **综合计算**：将多个 metrics 加权组合，进行聚类（clustering）或降维（如 t-SNE, UMAP, SOM）
- 多种方法可并行尝试

### 1.3 设计原则与验证

**实验驱动的设计理念**
- 不存在必须遵循的固定原则
- 重点是通过实验来验证和选择方法
- 多个实际 case 测试 layout 效果
- Layout 需要符合数据的内在性质
- Visualization 的组织方式应与 insight 的分类方式保持一致

**开发流程**
1. 先运行多个实际 case，找几份数据集
2. 通过 LLM 或自动化分析框架，生成大部分 insights
3. 模拟最终状态：数据集分析一段时间后的完整状态
4. 基于完整状态设计 layout 方案
5. 组合多种方法，提出最优设计

## 二、Progressive Visualization（研究贡献点）

### 2.1 Visual Continuity（视觉连续性）

**核心理念**
- 当新的 insight 加入时，layout 不应完全重新布局
- 需要采用渐进式（progressive）更新策略
- 保持视觉的连续性，让用户能够追踪 insight 的变化

**优化方向**
- 设计平滑的过渡动画
- 最小化现有元素的位置变化
- 让用户能够清晰地观察到新增 insight 及其与已有 insight 的关系

### 2.2 Progressive Update 实现方法

**渐进式传播策略**
1. 保持上一帧的 layout
2. 新 insight 进来时，先计算其大致位置（approximate position）
3. 影响逐渐传播（propagate），而非一次性重新布局
4. 让变化一点一点发生，实现平滑过渡

**Ideal vs Real Layout**
- **Ideal Layout**：基于当前所有 insights 计算的最优布局
- **Real Layout**：实际显示的布局，不会一步到达 ideal layout
- Real layout 随时间逐渐向 ideal layout 靠近
- 用户可以观察整个地图的演化过程

**Visual Indicators（视觉指示器）**
- 移动拖尾效果
- 连接线条显示元素从哪里移动到哪里
- 通过视觉编码（visual encoding）展示变化轨迹

### 2.3 Design Trade-off 思想

- 不存在唯一的 best solution
- 重点是提出一种设计，在特定场景下优于其他备选方案
- 科研工作就是找到相对最优的设计方案

## 三、Empty Space 的语义设计

### 3.1 地图隐喻的扩展

- 地图不仅展示已探索的 insights
- 还需要表明哪些空间未被探索

### 3.2 Empty Space 的意义

1. **Data Subspace 映射**
   - 未探索区域对应特定的数据子空间
   - 表示模型/分析尚未覆盖的区域

2. **潜在交互价值**
   - 可以给空白区域赋予语义
   - 用户可通过空白区域引导（steer）模型探索

3. **未探索方向的可视化**
   - 将 optional 探索方向纳入 layout
   - 替代传统 empty space，展示潜在的探索选择（choices）

### 3.3 Embedding Reverse 交互

- 如果使用 embedding 布局，可以反向推导空白区域的含义
- 用户在空白处点击时，通过坐标估计其对应的用户意图
- 作为 potential 的交互方式（非必须实现）

## 四、静态与动态可视化目标

### 4.1 静态目标（Static Layout）

**功能需求**
- 展示已探索和未探索的内容
- Insights 的整体分布
- 区分重要和不重要的 insights
- 提供层次化的 layout 结构（layout detail）

**类比形式**
- 类似阅读一份 insight report
- 快速了解哪些 insights 重要
- 数据中有什么有趣的模式（interesting patterns）

### 4.2 动态目标（Dynamic Visualization）

**功能需求**
- 从空白开始，展示逐步演化过程（evolve）
- 实时监控 agent 探索方向
- 判断是否朝着预期的 insights 探索
- 允许用户监控和引导（steer）探索过程

**开发优先级**
1. **先做静态**：建立完整的 layout 设计
2. **后做动态**：基于静态 layout 的 case
   - 一帧一帧变成动画
   - 调整 progressive update 策略
   - 添加探索方向的可视化

## 五、Visualization 设计核心理念

### 5.1 隐喻（Metaphor）的作用

**设计目标**
- 可视化要符合用户熟悉的事物
- 让用户一眼就能理解大致含义

**常见 Metaphor 例子**
- 汉堡菜单（食物隐喻）
- 地图（spatial metaphor）
- Tree（树状结构）

**地图隐喻的效果**
- 元素在 map 上分布
- 用户直觉理解为类似地图
- 具有一定的空间语义

### 5.2 Utility First 原则

**实用性优先**
- Metaphor 只是提供设计方向的指导
- 具体的 insight layout 和 presentation 需要符合实用性要求
- 能让用户看到重要 insights
- 能理解 agent 现在在做什么

**核心要求**
- 先满足 utility 需求
- 然后考虑 metaphor 的美观性

## 六、Interaction Design 研究方法

### 6.1 Initial Taxonomy 构建

**调研与扩充**
- 调研更多相关工作
- 总结形成 initial taxonomy（初始分类体系）

**验证方法**

1. **Expert Interview**
   - 邀请专家或有经验的人进行访谈
   - 通过 LLM 的形式完成分析任务
   - 观察（observe）他们在分析 agent output 时的行为

2. **Pilot Study**
   - 小规模试点研究
   - 观察用户的隐含行为模式

**观察要点**
- 用户进行的操作类型
- Think aloud 过程中的困惑点
- 需要进一步验证（verify）或澄清（clarify）的部分
- 实际的交互行为

### 6.2 Expert Interview 设计

**基础要求**
- 先建立 initial taxonomy
- 设计实验让用户产生目标行为
- 观察是否有更多 interaction 需求

**访谈起点**
- 基于上周讨论的 interaction 分类
- 看能否进一步扩充
- 访谈组内经常分析数据的人员

**交互模式转换**
- 传统 flow：直接与 LLM 对话
- 新模式：通过可视化界面与整体系统交互
- 需要探索新的交互方法

### 6.3 Interview 形式建议

**灵活的访谈方式**
- 不需要太 formal（正式）
- 找人聊一聊即可，集思广益
- 不要在完全没有准备的情况下直接去访谈专家
- 先有基础的设计方案，这样访谈才有方向

**两种可能的方式**

1. **让用户实际使用系统**
   - 前提：用户有足够的 LLM-based 数据分析经验
   - 用户了解 LLM 输出的 report 形式
   - 可能不需要实际操作系统

2. **展示设计方案**（推荐）
   - Present system prototype 或 design
   - 让专家提供想法和意见
   - 收集 initial interaction 需求
   - 了解超出考虑范围的需求

### 6.4 Prototype 准备

**设计稿制作**
- 用 Figma 等工具画 mock 图
- 基于理想中的 insight map
- 展示系统的大致样子

**展示内容**
- Interface 设计
- 可视化解释系统如何 work
- 希望专家反馈想要的功能

**Interface 设计要点**
- 确定需要哪些 view（视图）
- Interface 要符合最终系统设计
- 这样收集的建议才更有 constructive（建设性）
- 避免只收集到我们已经想到的建议
