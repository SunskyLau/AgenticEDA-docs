# 会议记录 - 2026-01-20

## 一、Insights Map 可视化设计

### 1.1 基础设计理念
- **节点规模**：节点数量控制在合理范围内
- **结合 Insight Taxonomy**：不同类型的 insights 映射到不同区域，相当于将高维空间投影为 insight map
- **关联关系**：insights 之间的线索关联类似于地图上的道路
- **动态边界**：地图边界根据探索过程动态变化

### 1.2 视觉编码策略
考虑基于 insights 的价值进行视觉编码：
- **节点大小**：反映 insight 的重要程度
- **距离中心的距离**：可作为另一种编码维度
- **高度维度**：讨论是否可以作为探索深度的编码
  - Di 的观点：更倾向于实现 overview to detail（zoom in / zoom out）
  - Shuhan 的观点：地图不一定局限于 2D，可以考虑 2.5D 或 3D

### 1.3 Focus in Context 机制
- **自适应显示**：大模型认为有深度的 insight 放大显示，甚至直接呈现对应的可视化；不太重要的 insight 以小节点形式呈现
- **交互探索**：用户可以 zoom in 查看小 insight 的详细内容

## 二、交互设计 (Interaction Taxonomy)

### 2.1 交互体系设计目标
建立系统化的探索交互体系，包括以下操作：
- **prefer**：优先探索某个方向
- **decline**：取消某个方向
- **provide information**：提供补充信息
- **accept**：接受建议
- **drill down**：深入挖掘
- **extend**：扩展探索
- **elaborate**：详细阐述
- **add node**：添加节点（plan/hypothesis）

### 2.2 人类知识的融入
- **必要时引入人类知识**：提供 text box 让用户 insert prompt
- **Clarify 机制**：为用户提供 input 手段，澄清和补充信息

### 2.3 资源管理与 Steering
- **问题背景**：与 LLM 的长链协作过程是 resource-intensive 的
- **Steering 策略**：
  - 优先分析什么：在 map 上圈选区块，优先探索该区域
  - 剪枝掉不好的方向：去除低价值探索路径
  - 设置约束 (constraint)：限定探索范围
- **目标**：将有限的资源用在最想了解的 insights 上

### 2.4 交互的时效性
- **实时性交互**：如剪枝操作，立即生效
- **滞后性交互**：如突然出现一个 sparkle，可能需要调整优先级，存在响应延迟

### 2.5 Insight 路径的深化
- **Path 补全**：从一个 insight 到另一个 insight 的 path，让系统 rethink 并添加中间节点
- **目的**：实现中间思路的深化和补全

### 2.6 Interaction Taxonomy 的构建方法
除了 brainstorm，还可以通过以下方法构建：
- Literature survey + 数据集分析 + brainstorm 形成初步的 taxonomy
- 然后与 expert 交流迭代
- 参考类似工作（如令雨的研究）：针对 running analysis agent 设计控制交互，然后构建系统将这些交互方法整合，结合 visualization design 形成系统化的设计

## 三、当前存在的问题

### 3.1 探索深度的问题
- 不同 insights 的深度差异较大（例如，有的 correlation insight 较深，有的较浅）
- 需要更好的机制来评估和展示探索深度

### 3.2 停止时机
- 讨论认为停止时机不是核心重要问题

## 四、研究定位 (Position)

### 4.1 核心问题
**人如何与运行过程非常长的大语言模型进行协作**

### 4.2 当前研究现状
- **现有方式**：主要是 one-shot 模式——让模型分析，查看结果，给出反馈，再继续分析
- **研究空白**：在推理过程中实时协作的长期流程探索较少
- **重要性**：这是一个重要且有待探索的问题

## 五、论文第三章计划

### 5.1 研究方法倾向
采用 VIS research 的经典形式：
- Expert interview
- Brainstorming session
- Requirement analysis
- 找几位 expert 定期沟通，定期设计迭代

### 5.2 用户研究设计
**两批用户的差异化研究策略**：

**第一批**：有真实 EDA 需求，但未必有专业 LLM 使用能力
- 进行长期研究
- 教授正确使用 LLM 的方法
- 观察使用过程中遇到的问题
- 反复测试和改进

**第二批**：有 LLM 使用能力，但没有 EDA 分析需求
- 补充不同角度的视角

**目的**：从写作角度支持一些设计假设 (design assumption)
